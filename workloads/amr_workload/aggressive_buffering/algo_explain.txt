Most optimistic g_cells buffering result
=======================================

The most optimistic graph is the isolated g_cells-only comparison, especially
the split DRAM-cycles plot. In that experiment, the original g_cells request
stream took 10,674,061 DRAMSim3 cycles, while the optimistic buffered version
took 2,339,212 cycles. That is about a 4.56x reduction in simulated DRAM
service time for this isolated stream. The bandwidth number moves by the same
basic factor, from 3.27698 to 14.9533, because the model is effectively asking
DRAMSim3 to service the same g_cells traffic in a much shorter time window.

The algorithm behind that optimistic graph is deliberately aggressive. It
extracts every request whose address falls inside the traced g_cells physical
range, then constructs a synthetic g_cells-only trace. The "original" version
keeps those requests in their original relative timing. The "optimistic" version
sorts the reads by address, emits them as one read phase, then sorts the writes
by address and emits them as one write phase. This removes almost all CPU-side
spacing and most read/write alternation from the g_cells stream, giving DRAMSim3
a best-case request presentation where buffering is effectively unlimited and
reordering is free.

The non-cycle graph is useful for sanity checking the mechanism. The completed
read and write counts are roughly the same between original and optimistic runs,
so the comparison is not just dropping most of the traffic. Average read latency
does not improve; it actually rises from 31.5006 to 34.2231 cycles. That is
compatible with the model: the main win here is not lower per-request latency,
but higher throughput from presenting a dense, phase-separated stream to DRAM.
Row-hit counts also fall in the optimistic run, which is a warning that simple
address sorting plus burst injection is not the same thing as an ideal
row-buffer-aware scheduler.

Caveats
-------

This graph is an upper-bound estimate, not an implementation result. It assumes
an unbounded buffer, zero discovery cost, zero enqueue/dequeue cost, free
sorting/reordering, and no CPU stalls while reads wait inside the buffer. It
also does not prove that every store is safe to delay; it relies on the stated
working assumption that once g_cells is written, that value is not read until
the next iteration.

The whole-program optimistic trace did not show a visible aggregate gain. The
reason is that, within the current m5_work_begin/end windows, the script found
only 108,774 rewriteable g_cells requests, all reads, while the full trace has
27,116,399 requests. Many g_cells requests are outside those work windows. So
the isolated g_cells-only graph answers "how much could the g_cells memory
stream benefit in the best case?", while the whole-trace graph says "this
current marker placement and whole-program trace do not yet expose that benefit
at application scale."

There is also a trace-address caveat. The workload prints virtual g_cells
addresses, while the DRAMSim3 trace uses low physical-style addresses. For the
current AMR binary, the experiment uses the ELF-backed translation
virtual_address - 0x400000, mapping g_cells from 0x405020..0x435020 to
0x5020..0x35020 in the trace. If the binary layout changes, this translation
should be rechecked.
