Aggressive g_cells Read/Write Buffering Algorithm
=================================================

Goal
----
Estimate an optimistic upper bound for DRAM performance if memory requests to the
AMR `g_cells` array could be aggressively buffered and phased by operation type.
The model is intentionally generous: it is meant to answer "how good could this
idea look if the buffering machinery were very effective?", not to model a
specific implementable microarchitecture.

Inputs
------
- A DRAMSim3 trace generated from gem5:
  - `addr op cycle opt_flag`
- The gem5 workload log or workload stdout containing lines like:
  - `GCELL idx=0 base=0x... addr=0x... offset=0x... sizeof_cell=48`
- The gem5 log containing iteration markers:
  - `ITER_BEGIN t=<id> tick=<tick>`
  - `ITER_END   t=<id> tick=<tick>`

Address range recovery
----------------------
1. Parse all `GCELL` lines.
2. Recover the virtual `g_cells` interval:
   - start = reported `base`
   - end = max(reported `addr` + `sizeof_cell`)
3. Convert it to the trace address domain.
   - gem5 packet traces in this repo use low physical-style addresses, while
     the workload prints virtual addresses.
   - The script tests candidate deltas, including `0` and `-0x400000`, and
     chooses the translated interval that matches the most trace requests.
   - For the current AMR binary this maps:
     - virtual `0x405020..0x435020`
     - trace `0x005020..0x035020`

Iteration windows
-----------------
1. Parse `ITER_BEGIN` and `ITER_END` ticks from the gem5 log.
2. Convert ticks to DRAMSim cycles with integer division by `--tick-div`, which
   defaults to `630` to match the existing conversion flow.
3. Treat each iteration as an independent reordering window.
4. Requests outside those windows are left unchanged. This avoids making claims
   about initialization and final checksum traffic.

Trace rewrite
-------------
For each iteration window:
1. Split trace entries into:
   - `g_cells` reads
   - `g_cells` writes
   - all other traffic
2. Leave all non-`g_cells` traffic at its original cycle.
3. Remove the original `g_cells` requests from their original cycles.
4. Reinsert `g_cells` requests as two dense phases at the beginning of the
   iteration:
   - read phase first
   - write phase second
5. Within each phase, sort by address to maximize row locality.
6. By default, emit every request in a phase at the same DRAM cycle. This is the
   most optimistic burst model: it assumes a large enough buffer to absorb the
   requests and present them to DRAMSim3 all at once.
7. Preserve the trace's optional fourth field (`opt_flag`) if present.

Why reads before writes?
------------------------
The intended software property is that `g_cells` writes do not feed later reads
until the next iteration. Under that assumption, the writes in an iteration can
be delayed past the reads from the same iteration. Grouping reads and writes
separately removes many read/write turnarounds from the request stream and gives
DRAMSim3 a best-case opportunity to exploit row locality within each phase.

What this does not model
------------------------
- Finite buffer capacity.
- CPU stalls while waiting for missing reads.
- Dependency checks beyond the per-iteration assumption above.
- Cache effects.
- Address alias/correctness hazards from stores that are actually read again in
  the same iteration.
- Any cost to discover, enqueue, sort, or drain the buffered requests.

Outputs
-------
- A rewritten DRAMSim3 trace.
- A JSON summary containing the chosen translated `g_cells` range, request
  counts, per-iteration counts, and cycle-span changes.
- DRAMSim3 result directories for baseline and optimistic traces.
- Matplotlib plots comparing baseline vs optimistic DRAMSim3 statistics.
